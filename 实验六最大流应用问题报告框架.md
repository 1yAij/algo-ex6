# 实验六《最大流应用问题：棒球赛问题》报告构思框架

> 本文件不是正式实验报告正文，而是用于正式写作前的实验分析、章节设计、图表规划和实验路线安排。正式报告可以在此框架基础上展开成 30 页左右的完整《深圳大学实验报告》。

## 一、实验整体定位

本次实验的主题是“最大流算法在应用问题中的建模与求解”，具体问题是经典的 Baseball Elimination，即棒球赛淘汰判定问题。

与前几次实验相比，本实验的重点不只是实现一个最大流算法，而是要完成一次完整的“应用问题到网络流模型”的转化。因此报告主线不应写成“介绍 Ford-Fulkerson，然后跑数据”，而应写成：

```text
现实竞赛问题
→ 表面胜场上限判断
→ 非平凡淘汰现象
→ 剩余比赛分配问题
→ 转化为最大流可行性问题
→ 用最大流/最小割定理证明正确性
→ 用四队样例和 Princeton 数据验证
→ 比较不同最大流算法实现效率
```

本实验的核心亮点应放在三个地方：

1. **建模能力**：如何把“未来比赛胜负分配”转化成流网络中的容量分配。
2. **理论证明**：为什么“满流”恰好等价于某球队仍有夺冠可能。
3. **算法优化**：在 Ford-Fulkerson、Edmonds-Karp、Dinic 等算法中选择更适合本问题的数据结构和实现方式。

## 二、实验问题分析

### 1. 问题背景

棒球赛季中，每支球队已经有若干胜场、负场和剩余比赛。判断某支球队 `x` 是否仍可能以第一名或并列第一名结束赛季，并不是只看 `w_x + r_x` 是否超过当前最高胜场。

最简单的淘汰情况是：

```text
w_x + r_x < w_j
```

即使球队 `x` 剩余比赛全胜，也无法追上某支已经胜场更多的球队 `j`，此时 `x` 被平凡淘汰。

但实验的核心在于**非平凡淘汰**。例如某些强队之间还要互相比赛，这些比赛必然会把胜场分配给其中一方，从而可能导致某支看似还有机会的球队实际上已经无法夺冠。

报告中应先用旧金山巨人队例子引入：

- 1996 年 9 月 10 日：巨人队最多只能达到 79 或 80 胜，明显追不上，因此被淘汰。
- 1996 年 9 月 8 日：巨人队表面上还能达到 81 胜，但 Padres 和 Dodgers 之间还有 7 场比赛，必有一方至少再赢 4 场，达到 82 胜，因此巨人队已经提前淘汰。

这段背景适合作为第一章开头，用来说明“局部胜场上限判断不够，必须考虑剩余赛程之间的相互制约”。

### 2. 输入数据含义

每支球队 `i` 有以下数据：

- `w_i`：当前胜场数；
- `l_i`：当前负场数；
- `r_i`：剩余比赛场数；
- `g_ij`：球队 `i` 与球队 `j` 之间还要进行的比赛场数。

对于待判断球队 `x`，它最多能达到的胜场上限为：

```text
M_x = w_x + r_x
```

如果其他球队最终胜场都能控制在 `M_x` 以内，则 `x` 仍有并列第一的可能；否则 `x` 被淘汰。

### 3. 四队样例数据

实验给出的四支球队为：

| Team | Wins | Losses | To play | vs Atl | vs Phi | vs NY | vs Mon |
|---|---:|---:|---:|---:|---:|---:|---:|
| Atlanta | 83 | 71 | 8 | - | 1 | 6 | 1 |
| Philly | 80 | 79 | 3 | 1 | - | 0 | 2 |
| New York | 78 | 78 | 6 | 6 | 0 | - | 0 |
| Montreal | 77 | 82 | 3 | 1 | 2 | 0 | - |

这组样例应作为全文第一个完整手算案例。

预期结论：

- Atlanta：未被淘汰。
- New York：未被淘汰。
- Montreal：被平凡淘汰，因为 `77 + 3 = 80 < 83`。
- Philly：被非平凡淘汰。虽然 `80 + 3 = 83`，表面上可追平 Atlanta，但其他球队之间的剩余比赛无法全部安排到不超过 83 胜的状态。

其中 Philly 是报告中的关键案例，因为它展示了最大流模型的必要性。

## 第二章 最大流算法基础与可行流思想

### 1. 章节目标

本章先建立最大流理论基础，为后续棒球赛网络构造和正确性证明做准备。

### 2. 应写内容

1. 流网络定义：
   - 源点 `s`；
   - 汇点 `t`；
   - 容量 `c(u,v)`；
   - 流量 `f(u,v)`。
2. 流量约束：
   - 容量限制；
   - 反对称性；
   - 流量守恒。
3. 最大流问题定义：

```text
maximize |f|
```

4. 残量网络与增广路径思想。
5. 最大流最小割定理。
6. 本实验中最大流的作用：
   - 不是求“最多能赢多少场”；
   - 而是判断“所有剩余比赛能否被合法分配”。

### 3. 可介绍的算法

报告中可以按以下层次介绍算法：

1. Ford-Fulkerson：思想最直观，适合解释增广路径。
2. Edmonds-Karp：使用 BFS 找最短增广路径，复杂度有保证。
3. Dinic：使用分层图和阻塞流，适合作为最终优化算法。

正式实现建议至少实现：

- Edmonds-Karp 作为基准最大流算法；
- Dinic 作为优化最大流算法。

如果时间有限，也可以只实现 Dinic，但报告中仍应解释它相对于普通 Ford-Fulkerson 的优化意义。

### 4. 复杂度分析方向

Edmonds-Karp：

```text
O(VE^2)
```

Dinic：

```text
O(EV^2)
```

在二分图结构或单位容量较多的网络中，Dinic 通常表现更好。

注意：这里的 `V` 和 `E` 是流网络中的点数和边数，不是棒球队数量。报告中要明确区分。

## 第三章 棒球赛问题的流网络构造

### 1. 章节目标

本章是报告核心之一，需要详细解释“为什么这样建图”。实验要求中的“解释流网络的构造原理”主要放在这里。

### 2. 对某支球队 x 的建模思路

假设要判断球队 `x` 是否被淘汰。为了让 `x` 尽可能不被淘汰，我们可以假设 `x` 剩余比赛全部获胜，因此它的最高胜场为：

```text
M_x = w_x + r_x
```

接下来只需要考虑除 `x` 以外的球队之间还剩下的比赛如何分配。

如果存在一种分配方式，使得每个其他球队 `i` 的最终胜场满足：

```text
w_i + assigned_i <= M_x
```

则球队 `x` 没有被淘汰。

否则，球队 `x` 被淘汰。

### 3. 网络节点设计

对每个待判断球队 `x`，单独构造一个流网络。

网络包含四类节点：

1. 源点 `s`。
2. 比赛节点 `G_ij`：
   - 对每一对 `i, j`，且 `i != x, j != x`，如果两队之间还有比赛，则建立一个比赛节点。
   - 该节点表示“球队 i 和球队 j 之间剩余的 `g_ij` 场比赛”。
3. 球队节点 `T_i`：
   - 对每个 `i != x` 建立一个球队节点。
   - 该节点表示球队 `i` 最终还能额外获得多少胜场。
4. 汇点 `t`。

### 4. 网络边与容量设计

边的设计如下：

#### 1. 源点到比赛节点

```text
s → G_ij, capacity = g_ij
```

含义：球队 `i` 和球队 `j` 之间还有 `g_ij` 场比赛，这些比赛必须全部产生胜者，因此这些“胜场资源”必须从源点流出。

#### 2. 比赛节点到球队节点

```text
G_ij → T_i, capacity = ∞
G_ij → T_j, capacity = ∞
```

含义：`i` 和 `j` 之间的每一场比赛，胜场可以分配给 `i`，也可以分配给 `j`。无限容量表示模型本身不限制具体分配比例，只由后面的球队胜场上限约束决定。

实际实现中，`∞` 可以设为所有剩余比赛总数，或一个足够大的整数。

#### 3. 球队节点到汇点

```text
T_i → t, capacity = M_x - w_i
```

含义：为了让 `x` 仍有机会并列第一，其他球队 `i` 最多只能再赢 `M_x - w_i` 场。如果该值为负数，则说明 `x` 已经被平凡淘汰。

### 5. 满流判定条件

设所有比赛节点从源点得到的总容量为：

```text
G_total = Σ g_ij, 其中 i != x, j != x, i < j
```

如果最大流满足：

```text
maxflow = G_total
```

说明除 `x` 以外所有球队之间的剩余比赛都能被合法分配，且没有任何球队最终超过 `M_x`，因此 `x` 未被淘汰。

如果：

```text
maxflow < G_total
```

说明至少有一部分比赛胜场无法在不超过 `M_x` 的条件下分配出去，因此 `x` 被淘汰。

## 第四章 最大流解决棒球赛问题的关键定理与证明

### 1. 章节目标

本章用于满足实验要求中的“证明该算法的关键定理，并说明这些定理所起的作用”。

### 2. 定理一：满流等价于存在合法赛程分配

可以命名为：

```text
定理 1：对于待判断球队 x，如果构造网络中的所有源点出边都能被完全饱和，则存在一种剩余比赛结果安排，使得 x 仍可能以第一名或并列第一名结束赛季。
```

证明方向：

1. 源点到比赛节点的容量 `g_ij` 表示 `i` 与 `j` 之间必须产生的胜场总数。
2. 如果这些边全部满流，说明每一场剩余比赛都被分配给了某个参赛球队。
3. 比赛节点只能把流量送到对应两支球队节点，因此不会出现把胜场分配给无关球队的情况。
4. 球队节点到汇点的容量是 `M_x - w_i`，因此球队 `i` 最多只能获得这么多额外胜场。
5. 所以每个 `i != x` 的最终胜场不超过 `M_x`。
6. 若 `x` 自己剩余比赛全胜，也能达到 `M_x`，因此 `x` 至少可以并列第一。

### 3. 定理二：无法满流等价于球队被淘汰

可以命名为：

```text
定理 2：如果最大流小于源点到比赛节点的总容量，则不存在任何一种剩余比赛结果安排能够使 x 保持第一名或并列第一名可能，因此 x 被淘汰。
```

证明方向：

1. 假设仍存在合法赛程分配。
2. 对每场 `i-j` 剩余比赛，将胜者对应的胜场作为从 `G_ij` 流向 `T_i` 或 `T_j` 的流量。
3. 因为所有比赛都产生胜者，所以源点出边应全部满流。
4. 因为合法分配要求每个球队最终胜场不超过 `M_x`，所以球队节点到汇点容量不会被超出。
5. 由此可构造一个满流。
6. 这与最大流不能满流矛盾，因此不存在合法安排，`x` 被淘汰。

### 4. 定理三：最小割给出淘汰证书

Princeton 规范中还要求给出 certificate of elimination，即淘汰证书集合 `R`。

可以写：

```text
定理 3：当球队 x 被淘汰时，最大流对应残量网络中从源点可达的球队节点集合，可以构成 x 被淘汰的一个证书集合。
```

证明方向：

- 根据最大流最小割定理，无法满流意味着存在容量不足的割。
- 割中对应的球队集合 `R` 表示这些球队之间的比赛和已有胜场共同形成了对 `x` 的胜场压力。
- 可以用不等式解释证书含义：

```text
(Σ_{i∈R} w_i + Σ_{i,j∈R} g_ij) / |R| > M_x
```

或者更直观地说：

```text
集合 R 中球队已经拥有的胜场，加上它们内部剩余比赛必然产生的胜场，使得至少有一支球队最终超过 x 的最高可能胜场。
```

## 第五章 算法设计与代码实现关系

### 1. 章节目标

### 2. 建议代码模块

正式实现可以分为以下模块：

#### 1. 数据读取模块

职责：

- 读取球队数量 `n`。
- 读取每支球队的名称、胜场、负场、剩余比赛。
- 读取 `n × n` 的剩余比赛矩阵。
- 建立球队名称到编号的映射。

报告中解释：

```text
该模块对应问题输入层，将 Princeton 数据格式转换为后续建图所需的数组结构。球队名称映射用于支持按名称查询某支球队是否被淘汰。
```

#### 2. 平凡淘汰检测模块

职责：

- 对待判断球队 `x`，计算 `M_x = w_x + r_x`。
- 若存在 `w_j > M_x`，则直接判定 `x` 被淘汰。

报告中解释：

```text
平凡淘汰是最大流建模前的快速剪枝。它对应球队节点到汇点容量可能为负的情况，可以在建图前直接判断。
```

#### 3. 流网络构造模块

职责：

- 为每个 `i, j != x` 建立比赛节点。
- 为每个 `i != x` 建立球队节点。
- 添加三类边：
  - 源点到比赛节点；
  - 比赛节点到球队节点；
  - 球队节点到汇点。

报告中解释：

```text
该模块是应用问题转化为最大流问题的核心。比赛节点负责表示必须分配的剩余胜场，球队节点负责接收这些胜场，而球队到汇点的容量则限制竞争对手不能超过 x 的最高胜场。
```

#### 4. 最大流模块

职责：

- 实现 Edmonds-Karp 或 Dinic。
- 返回最大流值。
- 保留残量网络，用于求最小割证书。

报告中解释：

```text
最大流模块并不知道棒球赛问题本身，它只负责在给定容量网络上寻找最大可行流。棒球赛语义全部体现在建图阶段。
```

#### 5. 淘汰判定模块

职责：

- 比较 `maxflow` 与 `G_total`。
- 若相等，则未淘汰。
- 若小于，则淘汰。
- 从残量网络中 BFS/DFS 找到从源点可达的球队节点，形成证书集合。

报告中解释：

```text
该模块对应最大流模型的判定定理。满流表示所有剩余比赛都能合法分配，非满流表示不存在合法安排。
```

## 第六章 四队样例求解过程

### 1. 章节目标

实验要求明确要求“给出上面四个球队的求解结果”。本章应对四队样例逐队分析，其中重点分析 Philly。

### 2. Atlanta

Atlanta 当前 83 胜，剩余 8 场，最高可达 91 胜。由于它本身是当前最高胜场球队之一，通常不会被淘汰。

结论：

```text
Atlanta is not eliminated.
```

### 3. New York

New York 当前 78 胜，剩余 6 场，最高可达：

```text
78 + 6 = 84
```

它仍可能超过 Atlanta 当前 83 胜，因此不属于平凡淘汰。通过最大流可验证存在一种剩余比赛分配，使所有竞争对手不超过 84 胜。

结论：

```text
New York is not eliminated.
```

### 4. Montreal

Montreal 当前 77 胜，剩余 3 场，最高可达：

```text
77 + 3 = 80
```

但 Atlanta 已经有 83 胜，因此：

```text
77 + 3 < 83
```

Montreal 被平凡淘汰。

结论：

```text
Montreal is eliminated by Atlanta.
```

### 5. Philly

Philly 当前 80 胜，剩余 3 场，最高可达：

```text
80 + 3 = 83
```

它表面上可以追平 Atlanta 当前 83 胜，因此不是平凡淘汰。

判断 Philly 时，排除 Philly，只考虑 Atlanta、New York、Montreal 之间的剩余比赛：

- Atlanta vs New York：6 场；
- Atlanta vs Montreal：1 场；
- New York vs Montreal：0 场。

这些比赛总共会产生：

```text
6 + 1 + 0 = 7
```

场胜利。

为了让 Philly 仍有机会，其他球队最终胜场都不能超过 83：

- Atlanta 已经 83 胜，因此还能再赢 `0` 场；
- New York 已经 78 胜，因此还能再赢 `5` 场；
- Montreal 已经 77 胜，因此还能再赢 `6` 场。

但是 Atlanta 与 New York、Montreal 之间还有 7 场比赛，只要 Atlanta 再赢任意一场，就会超过 Philly 的最高胜场 83。若 Atlanta 全输，则 New York 至少获得 6 场胜利，达到 84 胜，也超过 83。因此无论如何分配，都会有球队超过 Philly。

结论：

```text
Philly is eliminated.
```

这部分可以结合最大流网络展示：源点出边总容量为 7，但由于 Atlanta 球队节点到汇点容量为 0，流量无法全部合法送出，最终不能满流。

| Team | Max wins | Trivial eliminated? | Maxflow full? | Result | Certificate |
|---|---:|---|---|---|---|
| Atlanta | 91 | No | Yes | Not eliminated | - |
| Philly | 83 | No | No | Eliminated | 可由最小割给出 |
| New York | 84 | No | Yes | Not eliminated | - |
| Montreal | 80 | Yes | - | Eliminated | Atlanta |

## 第八章 最大流算法优化与复杂度分析

### 1. 章节目标

本章用于回应实验要求“尽可能实验优化的最大流算法”。

### 2. 可比较的算法路线

建议采用如下优化链条：

```text
Ford-Fulkerson 思想
→ Edmonds-Karp 基准实现
→ Dinic 分层图优化
→ 当前问题上的建图剪枝
```

### 3. 优化点

#### 1. 平凡淘汰提前判断

如果存在 `w_j > M_x`，不需要建图，直接返回淘汰。

作用：

- 减少不必要的最大流计算；
- 在大数据集上明显减少运行次数。

#### 2. 只为 `g_ij > 0` 的比赛建立节点

如果两队之间没有剩余比赛，不需要建立比赛节点。

作用：

- 减少网络节点数和边数；
- 对稀疏赛程数据更有效。

#### 3. Dinic 替代 Edmonds-Karp

Dinic 使用 BFS 构建分层图，再用 DFS 寻找阻塞流，通常比 Edmonds-Karp 更适合重复最大流计算。

#### 4. 每个球队单独建图，但复用基础数据

虽然每个待判断球队 `x` 都要建一个网络，但原始输入矩阵、球队编号映射、基础比赛对可以复用。

### 4. 流网络规模分析

设原始球队数量为 `n`。

对某支球队 `x`：

比赛节点最多有：

```text
C(n-1, 2) = (n-1)(n-2)/2
```

球队节点有：

```text
n - 1
```

再加源点和汇点，因此网络节点数最多为：

```text
V_flow = C(n-1,2) + (n-1) + 2
```

边数最多为：

```text
E_flow = C(n-1,2) + 2C(n-1,2) + (n-1)
       = 3C(n-1,2) + (n-1)
```

其中：

- 第一项是源点到比赛节点；
- 第二项是比赛节点到两个球队节点；
- 第三项是球队节点到汇点。

### 5. 建议实验

1. Edmonds-Karp vs Dinic 运行时间对比。
2. 是否启用平凡淘汰剪枝的运行时间对比。
3. 是否跳过 `g_ij = 0` 比赛节点的建图规模对比。
4. 不同 `n` 下流网络节点和边数量变化。
